# evidence-gate — 초안은 제출물이 아니라 검토 대상이다

## 0. 원칙 — 자기보고 무효

"확인했습니다"라는 문장을 생성하는 비용은 실제 확인 비용보다 훨씬 싸다. 그래서 진술은 품질 신호가 아니다. 모든 통과 판정은 아티팩트(명령 출력, exit code, diff, 파일:라인)가 내리고, 문장은 그 아티팩트를 인용하는 역할만 한다.

고정 지점은 두 개다. (1) 시작할 때 규모를 숫자로 판정한다(§1). (2) 초안 직후·보고 직전에 한 번, 초안을 남의 작업물처럼 증거만으로 검토한다(§3). 나머지 흐름은 평소대로 가볍게 진행한다.

late-entry 규칙: 이 스킬이 작업 도중이나 초안 완성 후에야 로드됐더라도 면제는 없다. 닻(§2)의 부재는 §3을 건너뛸 사유가 아니라 소급 작성 의무다 — 원요청 원문에서 요구 목록을 소급 구성하고, done-check를 그 시점에 §2의 사다리로 정한 뒤 보고에 `done-check 사후 선정`을 표기하고, §3을 그대로 수행한다.

## 1. 규모 판정 — "간단함"은 기분이 아니라 숫자다

**변경의 정의 (수단 불문).** Edit/Write/NotebookEdit만이 아니라 파일·상태를 변이시키는 모든 수단이 변경이다: Bash/PowerShell 리다이렉션·heredoc, 패키지 설치(lockfile 변이), git working tree·히스토리 조작, 스크립트 실행으로 파일 생성, 외부 리소스 변이. 상태를 변이시킬 수 있는 명령 실행은 면제 대상이 아니다.

**면제와 선언 의무.** 변경이 없는 작업(조회·설명·검색, 변이 없는 명령 실행)은 규모 판정을 생략하되, 면제도 한 줄 선언한다: `scale: 조회 — 파일·상태 변경 없음`. 선언 부재 자체가 위반이다. §4의 증거·어휘 규칙은 답변에도 적용된다. 특히 답변에 사용자가 적용할 의도의 코드/diff/명령을 담으면, 그 안의 이 워크스페이스 관련 식별자·경로·플래그는 §4 관찰 규칙의 대상이다 — 미관찰분은 미확인으로 표기하고 제안 전체에 UNVERIFIED 라벨을 붙인다.

**판정 단위.** 규모는 사용자 요청(턴) 전체 기준으로 한 번 판정한다. 하나의 요청을 여러 LIGHT로 분할 선언할 수 없고, 작업을 나눠 실행해도 규모 판정과 rubric은 요청 전체를 덮는다.

변경이 있으면 아래 다섯 조건을 센다. 전부 참이면 LIGHT, 하나라도 거짓이면 FULL. 경계가 애매한 조건은 거짓으로 센다(동률은 FULL).

- 변경할 파일이 1개다 (신규 생성 포함). 읽는 파일 수는 규모와 무관하다.
- 충족 여부를 따로 확인해야 하는 산출물이 1개다. 나열 토큰("그리고", "-고", "도")이 없어도 산출물이 2개면 2개로 세고, 변경에 따라오는 암묵 산출물(테스트 갱신, 문서 동기화)도 센다.
- 기존 함수 시그니처, 공개 동작, 설정 스키마를 바꾸지 않는다. 공개 동작이란 편집 파일 밖에서 호출·참조되거나 기존 테스트가 고정한 동작이다 — "internal이라서"는 면제 사유가 아니다.
- 비가역 작업이 없다 (삭제, `git reset --hard`, force push, 비용이 드는 외부 호출).
- 변경이 심볼 1~2개, 예상 diff 수십 줄 이내다. 시그니처를 보존하는 전면 재작성은 LIGHT가 아니다.

**선언 형식.** FULL은 한 줄: `scale: FULL — 파일 3개, 산출물 2개`. LIGHT는 과소 카운팅이 선언에서 드러나도록 적는다: `scale: LIGHT — 파일: <경로>, 요구: "<요청에서 인용한 구절>"`.

**TRIVIAL 변형.** 실행 의미가 없는 변경(주석, 문서 산문, 문자열 오타 등 비실행 텍스트)이고 파일 1개면, LIGHT에서 R2를 생략하고 편집 hunk의 diff 인용 한 줄을 증거로 보고한다. UNVERIFIED 라벨은 "실행됐어야 하는데 안 된 것"에만 쓴다 — 라벨의 신호 가치를 오타 수정으로 희석하지 않는다.

**래칫.** 규모는 상향만 가능하고 하향은 불가능하다. 승격 트리거: 두 번째 파일을 변경하게 될 때(읽기는 무관), diff가 예상 범위를 넘을 때, 편집 파일 밖 호출자를 발견했을 때, 조회로 시작한 작업이 변경으로 전환될 때(첫 변이성 툴 호출 직전에 규모 판정이 발동한다). 승격 시 건너뛴 닻을 소급 작성하되, 소급 done-check는 이미 쓴 코드가 아니라 원요청의 요구 문구에서 도출하고 보고에 `done-check 사후 선정 — 요구 n 기준`을 표기한다. "이미 반쯤 했으니 그냥 가자"가 이 스킬이 막는 패턴이다.

**규모 무관 고정 규칙.** (a) 비가역 작업 직전에는 복구 지점을 만든다: git 환경이면 `git stash create`(working tree를 건드리지 않는 스냅샷), 비 git이면 대상 파일의 백업 사본(작업 종료 시 정리 — 잔여물 규범). 사용자 요청 없는 자동 commit은 하지 않으며, commit이 더 적절하면 먼저 묻는다(harness git 규칙). (b) §2의 read-before-write와 §4의 증거 규칙은 LIGHT에도 적용된다.

## 2. 본 흐름 — 닻 네 개

FULL일 때 착수 시 네 가지를 적는다. 표가 아니라 몇 줄이면 충분하다.

1. **요구 목록** — 원요청에서 요구를 표현한 구절을 각각 인용하고 번호를 매긴다. 이 인용-번호 매핑이 §3 R0/R1의 행이 된다. "요청 처리" 같은 한 덩어리 분해는 R0에서 FAIL이다.
2. **done-check** — 이 변경이 잘못됐다면 무엇이 잡아낼지, pass/fail 신호를 낼 명령 하나를 코딩 전에 고정한다(끝에 가서 고르면 이미 한 작업이 통과할 체크를 고르게 된다). `없음`을 선언하려면 다음 사다리를 위에서부터 한 줄씩 기각해야 한다: (1) 프로젝트 테스트 러너 (2) build/compile (3) parse/lint 체크(`python -m py_compile`, `node --check`, PS parser) (4) import-only 실행·dry-run (5) 산출물 존재·형식 확인(`Test-Path`, schema validator). "테스트가 없음" 한 줄로는 불성립이다 — (3) 이하까지 불가능한 경우는 드물다. 사다리 검토 없는 `없음`은 R2 FAIL로 취급한다. 정당한 `없음`은 보고가 §4의 UNVERIFIED 규칙을 피할 수 없음을 미리 받아들이는 선언이다.
3. **범위 선언** — 변경할 파일 목록. 완료 시 R3에서 대조한다.
4. **결정점 목록** — 합리적 대안이 둘 이상 실재하는 갈림길(라이브러리 선택, 스키마 형태, 삭제 범위 등)을 전부 나열한다. 처리 방식은 메모리 ask-dont-assume 규범 그대로다: 추천안을 달아 일괄 질문하고, 명시적 go 전에는 해당 갈림길을 지나가지 않는다. "기본값 채택 후 통보"라는 범주는 없다. 기존 코드 관례를 그대로 따르는 무선택 지점(갈림길이 아예 없는 경우)만 결정점이 아니다.

편집 전에는 수정할 심볼의 정의를 시작부터 끝까지 Read하고, 시그니처나 동작을 바꾼다면 사용처를 Grep해서 영향 범위를 적는다. 파일 앞부분만 읽고 나머지를 추정으로 메우는 것은 편집이 아니라 환각이다. 읽지 않은 코드는 §3에서 증거가 될 수 없다.

plan mode가 활성이면 이 닻 네 개를 plan 문서 안에 포함시켜 제시하고, 승인 후 본 작업을 시작한다 — 닻을 별도로 중복 작성하지 않는다.

## 3. 자기검토 패스 — 초안 직후, 보고 직전

초안이 완성되어도 바로 보고하지 않는다. 생성자가 아니라 검토자로, 입력을 아티팩트로 제한하고 출력을 이진 rubric으로 제한해 다시 읽는다. 검토 입력은 세 가지뿐이며, 작업하며 생긴 "내가 한 것 같은" 기억은 입력이 아니다.

- **원요청 원문** — 다시 읽는다. re-read의 산출물은 R0의 인용-매핑이다(읽었다는 주장은 증거가 아니다).
- **diff** — `git diff` 출력. 비 git이면 편집한 hunk ± 주변 수십 줄만 다시 Read한다(파일 전체 재독 불요).
- **done-check 실행 출력** — 마지막 편집 이후 실행분. 없으면 지금 돌린다.

**신선도 규칙.** R2·R3·R4·R6의 증거는 마지막 파일 편집(Edit/Write/파일을 쓰는 명령) 이후에 일어난 도구 호출의 출력이어야 한다. 편집→재실행 순서가 한 번이라도 깨졌으면 이전 출력은 증거로 무효다.

**transcript 앵커.** 증거 셀의 인용은 이 세션 도구 호출 출력에 원문 그대로(verbatim) 존재해야 한다. 대응하는 도구 호출을 transcript에서 가리킬 수 없는 인용은 증거가 아니라 위조로 간주한다. rubric 표는 done-check 결과를 받은 그 턴에서 곧바로 작성한다 — 기억으로 나중에 채우지 않는다.

rubric은 전 항목 이진 판정 + 증거 셀이다. PASS의 증거 셀은 위치 참조만으로는 무효이고, 해당 아티팩트에서 복사한 라인 원문을 최소 1줄 포함해야 한다. 증거 셀이 비면 자동 FAIL — "확인했음"은 증거가 아니다.

| # | 항목 | 판정 | 증거 요건 |
|------|------|------|-----------|
| R0   | 요구 목록이 원요청의 모든 명시 요구를 포괄한다 | PASS/FAIL | 원요청 인용 구절 각각이 R1.n에 매핑 |
| R1.n | 하위 요구 n이 구현됐다 (요구마다 한 행) | PASS/FAIL | diff hunk의 변경 라인 원문 1줄 + 파일:라인 |
| R2   | done-check가 마지막 편집 이후 실행·통과했다 | PASS/FAIL/UNVERIFIED | 명령 원문 + exit code + 판정 라인 원문 |
| R3   | diff가 선언한 범위 안이다 | PASS/FAIL | `git diff --stat` 원문 / 비 git: 세션 변경 ledger |
| R4   | 변경이 참조하되 정의하지 않는 식별자·경로·플래그가 관찰됐다 | PASS/FAIL/N/A | 변경 밖 소스의 매치 라인 원문 |
| R5   | 시그니처·동작 변경의 사용처가 갱신됐다 | PASS/FAIL/N/A | Grep 결과 N건 + 처리 내역 |
| R6   | 새 함수·클래스·설정·핸들러가 호출·등록·연결됐다 | PASS/FAIL/N/A | 호출부 라인 원문 + 파일:라인 |

- R3: 비 git 환경의 증거는 선언 파일 re-Read가 아니라, 이 세션에서 파일을 변경한 모든 도구 호출(Write/Edit, 파일을 쓰는 명령 포함)의 대상 경로를 빠짐없이 열거한 목록(세션 변경 ledger)이다. 선언 외 파일이 diff/ledger에 있으면 기본값은 FAIL — 작업 중 그 시점에 범위 갱신을 transcript에 선언해 둔 경우에만 PASS이고, 사후 사유는 FAIL 판정과 함께 적는다.
- R4: 대상은 외부 참조다 — 기존 심볼, 라이브러리 API, CLI 플래그, 경로, 버전을 이 변경에서 작성하지 않은 소스(기존 코드, lockfile, `--help` 출력, 문서 fetch)에서 관찰한다. 방금 내가 만든 식별자를 방금 쓴 파일에서 관찰하는 것은 정보량 0 — 증거로 불인정. 이 diff에서 정의된 심볼은 R1/R6이 커버한다. 외부 참조가 없으면 N/A.
- R6: 잡는 것은 가장 조용한 실패 — 만들어 놓고 배선하지 않은 dead code다. 인용 라인이 def/class/등록 선언 자체이거나 주석·문자열 내부면 증거 무효. 호출부가 이번에 새로 만든 코드라면 기존 코드(이번 diff 밖) 또는 알려진 entry point에 닿을 때까지 호출 체인을 한 단계씩 따라가 각 단계를 인용한다.

**LIGHT 최소 보고.** 표 대신 다음을 넣는다: R2 한 항목(증거 포함; TRIVIAL은 diff hunk 인용으로 대체) + 범위 증거 한 줄(`git diff --stat` 원문 또는 세션 변경 ledger). 범위 증거에 파일이 2개 이상 나타나면 그 자체가 FULL 승격 트리거이며, 소급 닻 없이 보고하면 R3 FAIL과 동급이다. 추가로 새 심볼을 만들었으면 R6을, 이 세션에서 처음 참조한 외부 식별자·플래그·경로가 있으면 R4를 증거 포함으로 더한다 — 각각 Grep/Read 1회짜리 저비용 검사다.

**FAIL 처리.** 길은 두 갈래뿐이다: 고치거나, FAIL인 채로 보고서에 그대로 적는다. 항목을 조용히 빼거나 PASS로 바꾸는 세 번째 길은 없다. 고친 경우 해당 항목만 재검토하는 것이 아니라 — diff가 변했으므로 — 최소 R3와, 수정된 파일을 증거로 갖는 모든 행을 신선도 규칙대로 재실행한다. 확신 낮은 의심 사항도 전부 보고에 올리되 메모리 사실/추정 규범에 따라 '추정' 라벨을 명시한다 — 라벨 없는 의심은 사실 진술로 오독된다. 보고 가치의 필터링은 사용자의 일이다.

**검토 스코프.** 정확성과 명시된 요구사항으로 제한한다. 스타일 선호, "이왕이면" 개선, 요청 밖 리팩터링은 대상이 아니다. 발견 없음도 유효한 결과다 — 검토자라고 뭐라도 지적해야 하는 것이 아니다.

**cross-verify 점검 라인.** FULL 보고에는 rubric 아래에 `cross-verify: 해당 없음(§6 트리거 불충족)` 또는 `cross-verify: 제안 — <해당 트리거>` 한 줄을 반드시 적는다. 이 줄이 없는 FULL 보고는 불완전한 보고다.

## 4. 증거와 어휘 — 무엇이 "사실"인가

- **인용은 복사다.** 명령 원문, exit code, 판정 라인("3 passed, 1 failed")을 그대로 붙인다. 요약·재구성한 출력은 인용이 아니고, 기대한 출력을 실제 출력처럼 적는 것이 이 규칙이 막는 핵심 실패다. 인용이 먼저, 판정이 나중이다 — 이 순서면 실패 라인을 두고 성공이라 쓰는 모순이 transcript에 드러난다.
- **어휘는 의미 기준이다.** 런타임 동작을 단정하는 모든 서술은 — "검증됨" 계열이든 "반영했습니다", "이제 정상 동작합니다", "문제 없을 겁니다", "should work" 같은 동의·우회 표현이든, 한국어든 영어든 — 같은 보고 안의 실행 증거 인용이 있을 때만 쓸 수 있다. 단어 목록은 예시이지 회피 가능한 전체 집합이 아니다.
- **UNVERIFIED는 완료 선언이 될 수 없다.** R2가 UNVERIFIED인 보고는 첫 줄에 UNVERIFIED를 올리고, done/완료/됐다 계열 어휘 없이 "변경했으나 미실행 — 사유: ..." + 검증 방법 제안 + 사용자 결정 대기로 끝낸다. UNVERIFIED는 실패가 아니라 정직이지만, 정직이 완료를 대신하지는 않는다.
- **관찰 규칙의 스코프.** 이 워크스페이스의 파일·심볼·설정·실행 결과에 대한 서술은 이 세션 도구 출력에서의 관찰이 필요하다. 일반 기술 지식(표준 CLI 플래그, 라이브러리 API)은 메모리의 사실/추정 라벨 규범으로 서술하되, 그 답변에 근거해 비가역 작업이 실행될 상황이면 실측을 권한다.
- **done-check 치환 금지.** R2의 PASS는 §2에서 선언된 명령 원문의 실행에만 부여한다. 치환·축소했다면 사유와 함께 `PASS(축소)` 또는 UNVERIFIED로 구분 표기한다.
- **환경 각주.** diff·done-check 등 증거 수집 명령은 편집에 쓴 도구와 같은 경로 기준으로 실행하고, 불일치가 의심되면 Read로 재확인한다.

## 5. 막혔을 때 — 같은 문을 세 번 두드리지 않는다

같은 에러 메시지/증상을 향한 시도는 명령이 달라도 같은 접근이다 — 동일 증상 2회 실패가 2-strike이고, 미세 변형 세 번째 시도는 진단이 아니라 도박이다. 예외 하나: 에러가 명시적으로 일시성(timeout, lock, connection reset)을 가리키면 사유 한 줄과 함께 1회 추가 재시도를 허용하고, 그것도 실패하면 아래로 간다.

2-strike 후에는 진단 블록을 작성하고 실행한다: 에러 원문 인용, 서로 다른 가설 2개 이상, 가설을 가를 실험 1개. 실험은 적는 것이 아니라 실행하는 것이다 — 실험 출력 원문을 인용하고 어느 가설이 지지/기각됐는지 판정한 뒤에만 다음 수정을 시도할 수 있고, 다음 시도는 살아남은 가설을 명시적으로 인용해야 한다.

그래도 같은 지점에 막혀 있으면 외부 시선으로 간다: fresh-context Task subagent에게 문제 서술과 에러 원문만 주고 새로 보게 하거나, 사용자에게 막힌 지점을 보고하고 /cross-verify를 제안한다.

## 6. 외부 검증 — 자기 가중치의 맹점 밖으로

자기검토는 자기 가중치의 맹점을 벗어나지 못한다. 아래 신호가 하나라도 있으면 rubric 통과 후에도 /cross-verify(Codex 교차검증)를 사용자에게 제안한다(제안이지 자동 실행이 아니다). 점검 결과는 §3의 cross-verify 점검 라인에 남긴다.

- 비가역 작업이 포함된 변경 (마이그레이션, 대량 삭제, force push)
- 보안·비용·PHI를 다루는 코드 경로
- 3개 이상 파일에 걸친 구조 변경
- 자기검토의 FAIL을 고쳤지만 수정의 파급 범위가 확신되지 않을 때

사용자가 Codex 비용을 보류하면 무료 대안으로 fresh-context verifier subagent를 제안한다. verifier에게 주는 것은 원요청 전문, R# 요구 목록, diff, done-check 명령뿐이다 — 빌더의 추론과 변명을 공유하면 맹점도 같이 공유된다. (이 입력 제한은 무료 verifier에만 적용된다. /cross-verify는 자체 프로토콜이 라운드별 수정·반박 내역 공유를 요구하므로 그쪽을 따른다.) verifier의 "통과" 보고에도 §4를 그대로 적용한다: 인용 없는 통과는 기각하고 재실행을 요구한다.

무료 verifier가 회수한 finding의 처리는 세 갈래로 고정한다. (a) 재현을 먼저 시도하고, 재현되면 재현 증거를 첨부해 수정 여부를 사용자에게 묻는다(ask-dont-assume). (b) 기각하려면 반증 증거(코드 인용 또는 실행 출력)를 적는다 — "그럴 것 같지 않다"는 기각 사유가 아니다. (c) 정확성·명시 요구와 무관한 개선 제안은 고치지 않고 사용자에게 목록으로 전달한다. /cross-verify 실행 중이면 finding 처리는 그 스킬의 합의 루프(이슈별 내 견해 제시 → 사용자 합의 → 합의된 수정만 반영)가 우선한다.

## 7. 경계 — 기존 규범·다른 절차와의 관계

다음은 영속 메모리와 harness가 이미 커버하므로 재규정하지 않는다: 결정점 사용자 확인(ask-dont-assume — §2의 결정점 닻은 갈림길 감지를 아티팩트화할 뿐이다), 한국어 산문 + 영어 코드, 사실/추정 라벨(§4는 그 라벨의 판정 기준만 추가한다), 테스트 잔여물 정리. "검증 후 보고"는 harness의 일반 원칙이고, 이 스킬의 §3·§4는 그것을 폐기·대체하는 것이 아니라 판정 기준과 증거 형식으로 구체화한 것이다.

우선순위: 메모리·harness 지침과 실제로 충돌하면 그쪽이 우선한다. 단, 우선권을 발동하려면 메모리/지침의 해당 문장과 이 스킬의 해당 문장을 둘 다 인용해 실제 충돌을 보여야 한다 — 인용 없는 충돌 주장은 무효다.

다른 절차와의 분업:

- design-agent/implement-agent 워크플로우 실행 중에는 그 워크플로우의 자체 게이트·검증 계약이 절차를 지배하고, 이 스킬은 §4 증거·어휘 규칙만 적용한다(규모 판정·rubric 표를 중복 작성하지 않는다).
- 사용자가 /verify, /code-review, /cross-verify를 명시 호출하면 그 스킬의 절차가 우선한다.

## 부록 — FULL 작업 예시

착수 선언:

```
scale: FULL — 파일 3개, 산출물 3개
요구: 1) "X 버그 고치고"  2) "회귀 테스트도 넣고"  3) "CHANGELOG에 반영해줘"
done-check: pytest tests/test_x.py -q
범위: src/x.py, tests/test_x.py, CHANGELOG.md
결정점: retry 한도(3 추천 — 기존 관례와 일치) — 사용자 확인 대기
        삭제 동작 변경 여부 — 사용자 확인 대기
```

완료 보고 rubric (FAIL을 숨기지 않은 예; 인용은 전부 실제 도구 출력에서 복사):

```
| R0   | 요구 포괄      | PASS | "고치고"→R1.1, "테스트도"→R1.2, "CHANGELOG"→R1.3 |
| R1.1 | X 버그 수정    | PASS | src/x.py:142 `+    return cache.get(key, default)` |
| R1.2 | 회귀 테스트    | PASS | tests/test_x.py:88 `+def test_cache_miss_default():` |
| R1.3 | CHANGELOG 갱신 | FAIL | 증거 없음 — 누락됨, R3과 일치 |
| R2   | done-check     | PASS | `pytest tests/test_x.py -q` → exit 0, "12 passed in 1.31s" (마지막 편집 후 실행) |
| R3   | 범위 일치      | PASS | `git diff --stat`: "2 files changed" — CHANGELOG 미변경은 R1.3 FAIL로 보고 |
| R4   | 외부 참조 관찰 | PASS | 기존 심볼 cache: src/x.py:17 `from cache import cache` (Read) — diff는 호출만 추가 |
| R5   | 사용처 갱신    | N/A  | 시그니처 변경 없음 |
| R6   | wiring         | N/A  | 신규 심볼은 test 함수뿐 — 러너가 수집, R2가 증거 |
cross-verify: 해당 없음 (§6 트리거 모두 불충족)
```

## Remember

- 규모는 요청(턴) 전체에 대해 다섯 조건으로 한 번 정하고, 면제(조회)까지 포함해 반드시 선언한다. 상향만 가능, 하향 불가.
- 증거는 마지막 편집 이후의 도구 출력에서 verbatim 복사한 인용만이다. 증거 칸이 빈 항목은 FAIL이고, FAIL은 고치거나 그대로 보고한다 — 숨기는 선택지는 없다.
- UNVERIFIED 보고는 완료 선언이 아니다 — 검증 방법을 제안하고 사용자 결정을 기다린다. 새 코드는 호출부 인용이 있을 때만 연결된 것이다 (R6).
